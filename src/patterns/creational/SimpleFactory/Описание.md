
Порождающие шаблоны проектирования (Creational)

Простая Фабрика (Simple Factory)

Назначение

SimpleFactory в примере ниже, это паттерн «Простая Фабрика».

Она отличается от Статической Фабрики тем, что собственно не является статической.
Таким образом, вы можете иметь множество фабрик с разными параметрами.

Простая фабрика всегда должна быть предпочтительнее Статической фабрики!


Simple Factory (Простая фабрика) — это порождающий паттерн проектирования в объектно-ориентированном программировании (ООП),
который отвечает за создание других объектов и принимает решение о том, какой объект создать. 

Основная идея — инкапсулировать процесс создания объектов с помощью отдельного класса, 
чтобы клиент не знал деталей реализации этого процесса.

Основные характеристики

Инкапсуляция процесса создания объектов. Фабрика — это объект, который отвечает за создание и возвращение других объектов на основе входных параметров.

Отсутствие наследования. Simple Factory — это обычный класс с методом создания объектов, он не использует наследование.

Централизация логики создания. Логика создания объектов сосредотачивается в одном месте, что упрощает поддержку и тестирование кода.

Независимость клиента. Клиент получает объект через общий интерфейс, не зная конкретного класса, который будет создан.

Когда использовать

Simple Factory применяют, когда создание объекта включает определённую логику — не просто несколько присваиваний, а сложную логику. 
В этом случае имеет смысл создать отдельную фабрику вместо повторения одного и того же кода повсюду. 
Также этот паттерн подходит, если количество типов создаваемых объектов относительно постоянное и небольшое.

Преимущества

Снижение связности. Упрощает взаимодействие между клиентом и бизнес-уровнем, так как логика создания объектов скрыта от клиента.
Централизация изменений. Если структура создаваемых объектов изменится, нужно будет внести изменения только в класс фабрики,
а не во все места, где создавался объект.
Упрощение тестирования. Моковые реализации можно добавлять только в класс фабрики, а не во всём приложении.

Недостатки

Ограниченная гибкость. Привязка к конкретной реализации снижает гибкость системы. 
Simple Factory следует использовать только в тех случаях, когда архитектура не будет изменяться.

Ограниченность применения. Паттерн хорошо подходит, если количество типов объектов относительно постоянное.

Пример реализации

```php
class DoorFactory 
{
    public static function makeDoor(float $width, float $height): Door 
    {
        return new WoodenDoor($width, $height);
    }
}

// После этого можно использовать фабрику:

$door = DoorFactory::makeDoor(100, 200);
echo 'Ширина: ' . $door->getWidth();
echo 'Высота: ' . $door->getHeight();
```
