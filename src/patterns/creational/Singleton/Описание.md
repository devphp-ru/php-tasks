
### Порождающие шаблоны проектирования (Creational)

**Одиночка (Singleton)**

Это считается анти-паттерном! Для лучшей тестируемости и сопровождения кода используйте Инъекцию Зависимости (Dependency Injection)!

Назначение

Позволяет содержать только один экземпляр объекта в приложении, которое будет обрабатывать все обращения, запрещая создавать новый экземпляр.

Примеры

* DB Connector для подключения к базе данных
* Logger
* Config Manager
* Threads Handling
* Блокировка файла в приложении (есть только один в файловой системе с одновременным доступом к нему)

Singleton (с англ. «одиночка») — это паттерн проектирования, который гарантирует, что у класса будет только один экземпляр, и предоставляет к нему глобальную точку доступа. Это означает, что при попытке создать новый объект этого класса будет возвращён уже существующий экземпляр, а не создан новый. 

Принцип работы

Основные элементы паттерна:

* Приватный конструктор. Запрещает создание новых экземпляров класса.
* Статический метод, возвращающий единственный экземпляр. Обычно называется getInstance() или getSharedInstance().
* Статическая переменная для хранения экземпляра. При первом вызове метода экземпляр создаётся и сохраняется в этой переменной, при следующих вызовах возвращается уже существующий объект.

Когда используется

Паттерн применяется в ситуациях, когда требуется глобальное управление ресурсом или данными, например:

* Управление конфигурацией. Класс с настройками приложения (параметрами базы данных или внешнего вида интерфейса) обеспечивает единую точку доступа.
* Логирование. Единый объект логирования записывает все события в файл или базу данных.
* Подключение к базе данных. Гарантируется создание только одного экземпляра класса, отвечающего за взаимодействие с базой данных.
* Управление ресурсами. Например, единственный объект для взаимодействия с принтером.
* Генерация случайных чисел. Все компоненты программы используют один и тот же генератор.

Особенности реализации

* Ленивая инициализация. Экземпляр создаётся только при первом обращении к нему, что позволяет отложить создание до тех пор, пока оно действительно не потребуется.
* Потокобезопасность. В многопоточных приложениях может возникнуть проблема «гонки» между потоками, когда несколько потоков пытаются создать один и тот же экземпляр. Для решения используют механизмы синхронизации (например, блокировки).
* Использование декораторов (например, в Python). Позволяет «обернуть» класс и добавить дополнительную функциональность без изменения его структуры.

Преимущества и недостатки

Преимущества:

* гарантированное наличие единственного экземпляра класса;
* глобальная точка доступа к экземпляру;
* возможность отложенной инициализации.

Недостатки:

* Усложнение тестирования. Паттерн привносит в программу глобальное состояние, что затрудняет изолированное тестирование классов, зависящих от Singleton.
* Проблемы с многопоточностью. Доступ к глобальному объекту из нескольких потоков может привести к ошибкам синхронизации.
* Снижение гибкости. Изменение поведения Singleton может повлиять на все части приложения, которые его используют.
* Риск чрезмерного использования паттерна, что может привести к сильной зависимости между компонентами системы.
